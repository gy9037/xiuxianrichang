# V2.1 迭代优化需求

> 日期：2026-04-05
> 状态：待开发
> 来源：用户反馈

---

## OPT-1：愿望类型标签调整

**当前状态**：愿望类型有三种标签：个人、通用、团队

**改为**：
- "个人" → 改名为 "单人"
- "通用" → 去掉，不再使用
- "团队" 保留不变

**影响范围**：
- 后端：`wishes` 表的 `type` 字段，允许值从 `个人/通用/团队` 改为 `单人/团队`
- 前端：愿望创建页的类型选择、愿望池筛选按钮、愿望卡片上的类型标签
- CSS：`.wish-type-个人` 改为 `.wish-type-单人`，删除 `.wish-type-通用`
- 已有数据兼容：如果数据库中有 type='个人' 的记录，查询时兼容显示为"单人"；type='通用' 的记录视同"单人"处理

## OPT-2：自定义行为添加功能

**当前状态**：具体行为列表是预置的，用户无法添加新行为

**需求**：
- 在每个行为大类的具体行为列表末尾，添加一个 ➕ 按钮
- 点击后弹出输入框，用户填写：行为名称、品质判定模板（时长型/数量型/打卡型）、基础量（数量型需要填）
- 添加后，该行为对所有家庭成员永久可见
- 新增行为存入数据库，不再需要重复添加

**实现方式**：
- 新建 `custom_behaviors` 表：`id, family_id, category, name, template, base_quantity, created_by, created_at`
- 加载行为列表时，合并预置数据（behaviors.json）和数据库中的自定义行为
- 新增 API：`POST /api/behavior/custom` 添加自定义行为

## OPT-3：运动行为按身体部位分类

**当前状态**：身体健康类下的具体行为是扁平列表（跑步、跳绳、游泳...）

**改为**：身体健康类增加子分类（二级分类），按训练部位分组

| 子分类 | 包含行为（示例） | 品质模板 |
|-------|---------------|---------|
| 上肢 | 俯卧撑、引体向上、哑铃弯举 | 数量型 |
| 核心 | 仰卧起坐、平板支撑、卷腹 | 数量型/时长型 |
| 胸背肩 | 卧推、划船、肩推 | 数量型 |
| 下肢 | 深蹲、箭步蹲、蹬腿 | 数量型 |
| 综合/有氧 | 跑步、跳绳、游泳、骑车、散步、瑜伽 | 时长型 |

**实现方式**：
- `behaviors.json` 中身体健康类的数据结构从扁平数组改为按子分类分组的对象
- 前端行为选择时，先选子分类，再选具体行为
- 其他大类（学习、生活习惯、家务、社交互助）暂不加子分类，保持扁平
- 对应属性不变，都是体魄

## OPT-4：UI 布局优化

### 4.1 奖励页合并到背包
- 底部导航去掉"奖励"tab（从6个减为5个：角色、行为、背包、愿望、家庭）
- 背包页新增一个tab切换：道具 | 奖励
- 道具tab显示现有的道具背包内容
- 奖励tab显示待兑现和已兑现的奖励列表（即原奖励页内容）

### 4.2 角色主页趋势图
- 删除"最近7天修炼趋势"卡片，当前数据太少没有参考价值
- 后续数据积累后可以恢复

### 4.3 属性总和区域优化
- 当前"属性总和 0.0"标题过于简陋
- 改为：顶部显示角色名+境界badge（保持），下方用一个更紧凑的卡片展示属性
- 属性网格改为单列列表样式，每行：图标+属性名 | 当前值/上限 | 进度条
- 进度条占满一行宽度，视觉上更清晰

## OPT-5：用户标签暂时隐藏

**当前状态**：角色主页有用户标签选择区域，但标签目前只影响Boss属性分布权重

**改为**：
- V2.1 暂时隐藏用户标签区域
- 后续设计"体检"功能后，通过体检结果自动生成标签
- 后端标签相关的API和逻辑保留，只是前端不展示

## OPT-6：境界突破进度优化

**当前状态**：境界信息分散在两处：顶部的境界badge、底部的"下一境界"卡片

**改为**：
- 境界badge后紧跟一个简短的进度指示，例如：`练气一阶 ▸ 二阶（10/10）`
- 含义：当前练气一阶，升到二阶需要属性总和≥10，当前进度
- 如果可以突破，badge 变为金色闪烁效果，点击即触发突破
- 删除底部单独的"下一境界"卡片

## OPT-7：Boss 战力展示

**当前状态**：
- 愿望卡片只显示难度星级，没有Boss预估战力
- 挑战Boss后的战斗结果页有Boss战力，但进入挑战前看不到
- Boss的5属性分布在战斗结果中也没有展示

**改为**：
- 愿望卡片上显示Boss预估战力范围（根据难度系数和用户历史胜场计算）
- 进入挑战页（选择装备道具的页面）时，先生成Boss并显示Boss信息：名称、描述、总战力、5属性分布（柱状图或数值列表）
- 战斗结果页也展示Boss的5属性分布
- 这样玩家可以根据Boss属性分布决定装备哪些道具

**实现调整**：
- 当前 `POST /api/battle/start` 同时生成Boss和战斗，需要拆分为两步：
  - `POST /api/battle/prepare` — 生成Boss，返回Boss信息，不开始战斗
  - `POST /api/battle/execute` — 传入 boss_id 和 equipped_item_ids，执行战斗
- 或者在前端，先调用一个预览接口估算Boss战力，展示后再确认开战

## OPT-8：错误提示优化

**当前状态**：部分错误提示不够明确或使用英文

**修复项**：
- 登录时账号不存在：当前显示"登录已过期"或不明确的错误。改为"用户名或密码错误"（出于安全不区分用户名不存在和密码错误，但提示要用中文且明确）
- 检查所有 API 返回的 error 字段，确保全部使用中文
- 前端 `api.js` 中的 fallback 错误信息 `'请求失败'` 保留，但确保网络错误（fetch 抛出异常）也有中文提示，如"网络连接失败，请检查网络"
- 401 状态的处理：当前跳转登录页时显示"登录已过期"，如果用户本来就没登录过（token无效），应显示"请先登录"
