# UX Review — 家庭页 (family.js)

> 审查日期：2026-04-09
> 用户群：家庭（含老人小孩），移动端为主
> 审查维度：操作步骤、即时反馈/loading、视觉引导、容错性、移动端适配、状态一致性

---

## 问题清单

页面加载时无任何 loading 状态，三个并发请求期间页面空白，老人/小孩容易误以为卡死 | 🔴 | `load()` 开始时立即渲染骨架屏或 spinner，请求完成后替换内容

点击表情按钮后触发 `this.load()` 全量重渲染，整个页面闪烁重绘，动态列表滚动位置丢失 | 🔴 | 改为乐观更新：点击后立即更新按钮状态，后台静默同步，失败时回滚并 toast 提示

`react()` 点击后无任何视觉反馈，按钮不禁用，快速连点会发出多个重复请求 | 🔴 | 点击后立即禁用按钮并加轻微动画，防抖或加 `loading` flag 防重复提交

三个 API 请求任意一个失败只 toast 一条 `e.message`，页面渲染被完全跳过，用户看到空白页不知道哪里出错 | 🔴 | 对三个请求分别容错，单个失败时降级展示该模块的错误占位，其余模块正常渲染

表情按钮 `padding:2px 10px` 触控区域过小，老人手指难以精准点击（低于 44×44px 最小触控标准） | 🔴 | 将按钮最小高度设为 44px，padding 改为 `8px 14px`

`feed-time` 使用 `toLocaleString()` 输出完整日期时间字符串，在小屏设备上容易换行或截断，且对老人不够直观 | 🟡 | 改为相对时间（"3小时前"、"昨天"），超过7天再显示具体日期

团队愿望进度中 `teamProgress` 成员状态直接输出原始英文/字段值（如 `pending`、`in_progress`），对家庭用户不友好 | 🟡 | 统一映射为中文标签："待挑战" / "进行中" / "已完成"，与上方 `w.status` 的处理保持一致

家庭成员列表无点击交互，但视觉上与可点击的 feed 动态行样式相近，容易让用户误以为可以进入详情 | 🟡 | 若成员行不可点击，去掉 hover 态；若需要跳转，补充明确的箭头图标或跳转逻辑

`members.length === 0` 时空状态文案"暂无其他家庭成员"措辞模糊，用户不知道如何邀请 | 🟡 | 改为带引导的空状态："还没有家庭成员，去邀请家人一起修炼 →"，附带跳转入口

表情按钮的 `title` 属性（如"灵气充沛"）在移动端不会显示，标签含义对新用户不透明 | 🟡 | 在按钮下方或 tooltip 中补充文字标签，或首次使用时展示引导说明

`quality` 字段若为非预期值会静默回退为"凡品"，用户看到的品质与实际不符，且无任何提示 | 🟡 | 保留回退逻辑，但在开发环境 console.warn 异常值，便于排查数据问题

页面无下拉刷新（pull-to-refresh）支持，移动端用户习惯下拉更新，当前只能退出重进 | 🟢 | 接入下拉刷新手势，触发 `FamilyPage.load()`

`feed-avatar` 只取名字首字，若名字为空则显示"?"，缺少背景色区分，多成员时头像视觉上无法快速区分 | 🟢 | 根据用户 id 或名字哈希分配固定背景色，提升辨识度

团队愿望进度区块在愿望较多时无高度限制，页面可能过长，移动端需要大量滚动才能看到全部内容 | 🟢 | 超过3条时折叠，底部加"查看全部"展开按钮
