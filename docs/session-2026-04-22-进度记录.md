## 二、Batch 1-3 待修复问题清单

### 🔴 必须修复（6个）

#### 问题1：POST /behavior 没有事务保护
- 位置：`server/routes/behavior.js`，POST /behavior 处理函数
- 问题：依次执行 6 次写操作（INSERT behaviors → INSERT items → UPDATE behaviors(item_id) → UPDATE characters(last_activity) → INSERT/UPDATE shortcuts → doCheckin），没有外层事务。任何一步失败都会导致数据不一致。
- 修复：用 `db.transaction()` 包裹整个 POST /behavior 的写操作流程。doCheckin 内部已有事务，需要改为 savepoint 嵌套。

#### 问题2：getStreak 逐日查询性能隐患
- 位置：`server/services/checkinService.js`，getStreak 函数
- 问题：用 while 循环逐日向前查 checkins 表。连续签到 365 天就执行 365 次 SELECT。
- 修复：改为单次查询取最近 N 条记录，内存中判断连续性：
```sql
SELECT checkin_date FROM checkins
WHERE user_id = ? AND checkin_date <= ?
ORDER BY checkin_date DESC LIMIT 365
```
然后在 JS 中遍历判断日期是否连续。

#### 问题3：目标达成通知用 setTimeout 800ms 等待异步请求
- 位置：`miniprogram/pages/behavior/behavior.js`，行为提交成功后的目标达成检查
- 问题：`loadMonthlyGoals()` 是异步的，用 setTimeout 800ms 等待不可靠，网络慢时 goals 还是旧数据。
- 修复：让 `loadMonthlyGoals` 返回 Promise，在 `.then()` 中做达成检查。

#### 问题4：behavior_goals 的 sub_type 是自由文本
- 位置：`server/routes/behaviorGoal.js`，POST /behavior-goal
- 问题：用户自由输入 sub_type，如果文本与实际行为记录不完全匹配，目标永远无法达成。
- 修复：POST /behavior-goal 时验证 sub_type 是否存在于当前用户可用的行为列表中（behaviorConfig + custom_behaviors）。前端改为下拉选择而非文本输入。

#### 问题5：pin-behavior 接口没有验证行为是否存在
- 位置：`server/routes/character.js`，PATCH /character/pin-behavior
- 问题：用户可以传入任意 category + sub_type 组合，后端只检查非空，没有验证行为是否真实存在。
- 修复：加一步验证，检查 behaviorConfig 或 custom_behaviors 中是否存在该行为。

#### 问题6：前端样式大量重复
- 位置：`miniprogram/pages/` 下 home.wxss、behavior.wxss、inventory.wxss
- 问题：`.modal-mask`、`.modal-content`、`.goal-progress-bg` 等样式在多个页面重复定义。
- 修复：抽取到 `app.wxss` 作为全局样式。

### 🟡 建议修复（6个）

#### 问题7：UTC+8 时间逻辑分散且不统一
- 位置：checkinService.js、character.js（JS 层手动加 8 小时）vs behavior.js、behaviorGoal.js（SQL 层用 'localtime'）
- 问题：JS 层和 SQL 层的时区处理方式不同，如果服务器时区不是 UTC+8 会出现不一致。
- 建议：抽取到共享的 `server/utils/time.js`，统一时区处理方式。

#### 问题8：GET /character 接口过重
- 位置：`server/routes/character.js`
- 问题：一个 GET 请求触发十几次数据库查询（character、user、cultivation、decay、realm、promotion、tags、trend、checkin、pinnedBehaviors、behaviorGoals）。
- 建议：后续考虑拆分为按需加载（/character/basic、/character/goals 等）。当前用户量极小，不紧急。

#### 问题9：月度目标查询逻辑重复
- 位置：`server/routes/character.js` 和 `server/routes/behaviorGoal.js`
- 问题：getCurrentPeriodKey + 查 goals + 查 counts + 构建 map 逻辑完全重复。
- 建议：抽取到 `server/services/behaviorGoalService.js`。

#### 问题10：道具描述 JSON 缺少启动校验
- 位置：`server/data/item-descriptions.json` 和 `server/data/items.json`
- 问题：新增道具名但忘更新描述时，generateItem 返回空描述，不会报错但体验不完整。
- 建议：启动时校验两个文件的 key 是否一致，打印缺失项。

#### 问题11：跨页面传参用 globalData
- 位置：`miniprogram/pages/` 多处
- 问题：用 `getApp().globalData.inventoryTab = 'rewards'` 跨页面传参，全局可变状态可能残留。
- 建议：统一用 EventChannel 或页面间通信机制。

#### 问题12：Codex 指令中 api.patch/delete 签名笔误
- 位置：codex-v30-batch3-frontend.md 中的代码片段
- 问题：`this.request(url, 'PATCH', data)` 参数顺序写反了，实际 api.js 中是 `this.request('PATCH', url, data)`。
- 说明：实际代码中 api.js 已有正确的 patch/delete 方法，此问题仅存在于指令文档中。如果 Codex 严格按指令执行会写出 bug。需要修正指令文档或确认实际代码是否正确。
