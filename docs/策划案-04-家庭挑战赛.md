## 一、背景与设计目标

### 为什么做家庭挑战赛

家庭愿望系统中，愿望由家庭成员共同发起、共同完成 Boss，具备协作属性。但当前缺少**竞争性**或**约定性**的家庭互动机制——没有一个方式让家庭成员彼此约定"我们来比一比"或者"大家一起坚持某件事"。

家庭挑战赛的目标：
1. **提供自定义的周期性挑战**：家庭成员可以主动发起，约定一段时间内完成某项行为目标
2. **道具质押机制增加承诺感**：质押道具参与，让挑战不只是口头约定，而是有代价的承诺
3. **合作与竞争共存**：根据发起方设置，可以是协作达成目标，也可以是相互竞争

## 二、挑战赛机制设计

### 2.1 发起挑战赛

任意家庭成员均可发起。发起时需设定以下内容：

| 配置项 | 说明 | 是否必填 |
|--------|------|---------|
| 挑战名称 | 挑战的标题，如"本月跑步王" | 必填 |
| 挑战内容 | 描述需要完成的任务（见2.2自动结算类型） | 必填 |
| 挑战周期 | 起止日期，最短 3 天，最长 30 天 | 必填 |
| 参与质押 | 参与者需质押的道具类型、品质、数量 | 必填 |
| 奖励道具 | 完成挑战可获得的道具类型、品质、数量 | 必填 |
| 邀请对象 | 选择邀请的家庭成员（可多选） | 必填 |

**关于奖励来源**：奖励道具由发起方从自己的库存中质押提供，不由系统凭空生成。发起方在发起时即消耗这些道具，如挑战未达到最低参与人数则退回。

### 2.2 自动结算的挑战类型

第一版只支持可自动结算的三种类型：

| 类型 | 说明 | 配置示例 |
|------|------|---------|
| **行为次数** | 挑战周期内，完成指定类别行为 N 次 | 本周完成体魄类行为 5 次 |
| **连续天数** | 在挑战周期内保持指定 sub_type 连续打卡 N 天 | 连续 7 天记录「早起」 |
| **属性积累** | 挑战周期内，指定属性临时值积累超过 N | 本月心性临时属性积累 ≥ 15 |

> 注：自由文本类型（自行描述任务）留待后续版本，需要引入人工或家庭互认确认机制。

### 2.3 最低参与人数

- 挑战赛**最少需要 2 人参与**（可包含发起方自身）
- 发起后 **24 小时**内，若参与人数不足 2 人：挑战发起失败，发起方质押的奖励道具退回
- 超过 24 小时后不再接受新参与者

### 2.4 参与流程

1. 受邀成员收到挑战赛通知，查看挑战详情
2. 确认参与：从库存消耗指定质押道具，正式加入
3. 若库存不足，无法参与该挑战

### 2.5 结算规则

挑战周期结束后，系统自动结算：

| 情况 | 结果 |
|------|------|
| 参与者完成挑战目标 | 获得发起方设定的奖励道具；质押道具**不退回**（视为挑战成本，消耗掉） |
| 参与者未完成挑战目标 | 质押道具**消耗**，无额外奖励 |

> **质押道具始终消耗**（无论成功失败），这是参与挑战的真实承诺成本，让挑战有重量感。
> 奖励道具来源于发起方的事先质押，发起方"出题"的代价是真实的库存消耗。

### 2.6 多人场景下的奖励分配

| 挑战模式 | 奖励分配方式 |
|---------|------------|
| 竞争型（先完成者获胜） | 发起方的全部奖励道具归**第一个完成目标的参与者** |
| 合作型（所有人都需完成） | 每位完成目标的参与者都获得一份奖励道具（奖励道具按参与人数倍数质押） |

> 发起方在设置时需选择模式（竞争型 / 合作型），并据此准备足量的奖励道具。

## 三、展示设计

### 3.1 家庭页「挑战赛」入口

在家庭页新增「挑战赛」模块：

```
┌────────────────────────────────────┐
│ ⚔️  家庭挑战赛                      │
│                                    │
│ 进行中  [本月跑步王]  剩余 5 天      │
│         你：3次 / 目标5次  ▓▓▓░░   │
│         小明：5次 / 目标5次  ▓▓▓▓▓  │
│                                    │
│ [+ 发起新挑战]                     │
└────────────────────────────────────┘
```

### 3.2 挑战结算通知

挑战结束时，向所有参与者推送结算结果：

> 「本月跑步王」挑战已结束。小明以 5/5 次率先完成，获得良品道具 ×2。你完成 3/5 次，质押道具已消耗。

### 3.3 家庭动态

挑战赛发起、他人完成目标、挑战结算，均在家庭动态中展示相关公告。

## 四、数据库设计

### 新增表：`family_challenges`（挑战赛主表）

```sql
CREATE TABLE family_challenges (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  family_id INTEGER NOT NULL,
  initiator_id INTEGER NOT NULL,
  name TEXT NOT NULL,
  challenge_type TEXT NOT NULL,   -- 'behavior_count'|'streak'|'attr_accumulation'
  challenge_config TEXT NOT NULL, -- JSON：具体目标参数
  mode TEXT NOT NULL,             -- 'competitive'|'cooperative'
  stake_config TEXT NOT NULL,     -- JSON：质押要求 {category, quality, count}
  reward_config TEXT NOT NULL,    -- JSON：奖励内容 {category, quality, count}
  start_at DATETIME NOT NULL,
  end_at DATETIME NOT NULL,
  status TEXT DEFAULT 'recruiting', -- 'recruiting'|'active'|'completed'|'failed'
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 新增表：`challenge_participants`（参与记录表）

```sql
CREATE TABLE challenge_participants (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  challenge_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  stake_item_ids TEXT,            -- JSON：质押的道具 ID 列表
  progress INTEGER DEFAULT 0,    -- 当前进度值
  is_completed INTEGER DEFAULT 0,
  completed_at DATETIME,
  reward_issued INTEGER DEFAULT 0,
  UNIQUE(challenge_id, user_id)
);
```

## 五、后端改动清单

| 新增路由 | 说明 |
|---------|------|
| `POST /api/challenge` | 发起挑战赛（含奖励道具质押验证） |
| `POST /api/challenge/:id/join` | 参与挑战赛（消耗质押道具） |
| `GET /api/challenge/family` | 获取家庭内所有挑战赛列表 |
| `GET /api/challenge/:id` | 获取挑战赛详情及各参与者进度 |
| `POST /api/challenge/:id/settle` | 手动触发结算（也可定时任务自动） |

| 定时任务 | 说明 |
|---------|------|
| 每日 00:30 | 扫描所有 `end_at < NOW()` 且 `status = 'active'` 的挑战赛，执行自动结算 |
| 每日 00:30 | 扫描所有 `recruiting` 且超过 24 小时未达到 2 人参与的挑战赛，标记为 `failed`，退还发起方奖励道具 |

## 六、待确认问题

1. 发起方是否默认参与自己发起的挑战？建议：可选，发起时决定，若参与则也需质押道具
2. 竞争型挑战中，若第一名完成后，其他人是否继续计时？建议：继续，直到周期结束，其他人完成了也算达标（但奖励已被领取，仅避免质押损失）
3. 单人参与（发起人 + 1 人）时，合作型是否有意义？建议：有，相当于两人约定共同完成一件事，奖励各自获得
