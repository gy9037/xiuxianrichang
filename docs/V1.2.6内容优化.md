# V1.2.6 内容优化方案

> **版本**：V1.2.6 → V1.3.0
> **日期**：2026-04-16
> **来源**：微信开发者工具截图审查 + 代码走读
> **范围**：系统设计、UI 设计、交互设计、新增功能建议

---

## 〇、核心问题诊断

产品的体验循环是：

```
记录行为 → 获得道具(临时属性) → 炼化道具(永久属性) → 挑战愿望(消耗道具) → 达成目标(获得奖励)
```

当前五个页面各自展示数据，但**用户感知不到这条链路**。每做一步都不知道对下一步有什么影响。以下所有调整方案都围绕一个目标：**让用户在每个操作节点都能感知到因果关系和前进方向。**

## 一、首页（角色页）

### 1.1 角色身份区

| | 现有设计 | 调整方案 |
|---|---|---|
| **角色名** | `font-size:40rpx` 纯文字，和右侧"居家"状态标签同行，无修仙代入感 | 角色名保持大字，下方增加一行**境界意境描述**（如练气一阶："初窥天道，灵气尚薄"），由后端按 realm_stage 返回，纯展示不可编辑 |
| **状态标签** | 绿色胶囊"居家 ▾"，悬浮在右上角，和境界信息视觉上无关联 | 移到角色名正下方，与境界徽章同行排列：`[练气一阶] [居家]`。两个胶囊并排，左为境界（渐变），右为状态（纯色） |
| **修炼状态** | 境界徽章、修炼等级(懈怠)、天数(3/7天)、类别数(2类) 挤在一行 wrap 排列，信息密度过高但层级全平 | 拆为两层：第一层只展示**境界 + 修炼等级**（如"练气一阶 · 懈怠"）；第二层展示活跃天数和类别数，用更小的字号和灰色 |

### 1.2 修炼进度引导（新增模块）

| | 现有设计 | 调整方案 |
|---|---|---|
| **nextLevelHint** | "再活跃1天即可达到稳修"——24rpx 灰色小字，放在角色卡底部，极易被忽视 | **升级为独立卡片**，置于角色卡下方、属性总览上方。包含：(1) 一句话提示文案；(2) 进度条（当前活跃天数 / 目标天数的可视化填充）；(3) 右侧"去修炼 →"引导按钮，点击跳转行为页。这是首页最重要的行动驱动元素 |
| **加成/惩罚显示** | "良品+10%"和"缓冲-5天"作为文字标签散落在修炼状态行末尾 | 合并到修炼进度卡片中，作为进度条下方的辅助说明：当有加成时显示"当前增益：良品掉率 +10%"（绿色），有惩罚时显示"当前衰减：缓冲 -5 天"（红色） |

### 1.3 属性总览

| | 现有设计 | 调整方案 |
|---|---|---|
| **雷达图** | 始终渲染 canvas 雷达图（500rpx × 400rpx），属性全为 0 时显示一个扁平的最小多边形，视觉上无意义且占半屏空间 | **分两种状态展示**：① 全属性为 0 时，隐藏雷达图，展示空状态引导："完成行为 → 获得道具 → 炼化提升属性"，配合简单的流程示意图（三个步骤图标横排）；② 任一属性 > 0 时，正常展示雷达图 |
| **属性标签位置** | 雷达图外圈的属性名 + 数值用 canvas drawText 绘制，字号 12px/11px，在小程序中渲染偏小 | canvas 内标签字号增大至 13px/12px；或改为 canvas 只画图形，属性名和数值用 WXML 元素覆盖在 canvas 四周，便于控制字号和点击 |

### 1.4 修炼趋势

| | 现有设计 | 调整方案 |
|---|---|---|
| **数据不足时的展示** | 即使只有 1-2 根柱子也完整展示 7 天图表区域，大量空白柱 | 活跃天数 < 3 天时，**折叠整个趋势卡片**，仅显示一行文字："修炼满 3 天后解锁趋势分析"（可点击展开查看）。≥ 3 天后正常展示 |
| **图例** | 右侧竖排5个色块（16rpx × 16rpx），每个只显示一个汉字（体/悟/心/灵/神），辨识度低 | 图例移到图表下方横向排列，色块增大至 20rpx × 20rpx，显示完整名称（体魄/悟性/心性/灵巧/神识） |

### 1.5 今日推荐

| | 现有设计 | 调整方案 |
|---|---|---|
| **卡片标题** | "✨ 今日推荐"——功能性准确但缺乏紧迫感 | 改为带动态内容的标题：若类型为 first_behavior，标题为"开始你的修炼之路"；若为 synthesize，标题为"道具已就绪"；若为 weak，标题为"短板提醒" |
| **推荐项文案** | "道具不够合成（需同属性累计10点），继续积攒"——信息准确但偏系统语言 | 改为更具体的量化引导："灵巧道具临时值 7/10，再获得 3 点即可炼化"——告诉用户差多少、差什么 |

## 二、行为页

### 2.1 快捷打卡区

| | 现有设计 | 调整方案 |
|---|---|---|
| **布局** | `grid-template-columns: 1fr 1fr`，按钮高度 `min-height: 72rpx`，按钮内 `flex-direction: row`（名称和次数横排） | 截图中按钮实际视觉偏高偏空。调整为：按钮内改为**竖排**（名称在上，次数在下），缩小按钮内 padding 为 `12rpx 20rpx`，让4个按钮更紧凑地出现在一屏内 |
| **次数显示** | 显示历史总次数，如"3次" | 改为**今日维度**："今日 2 次"或无打卡时显示"今日 0 次"（灰色），让用户对今天的进度有感知 |
| **已完成状态** | 无任何视觉标记，打过卡和没打过的按钮外观完全相同 | 今日已打卡的按钮左上角加一个小圆点标记（紫色，8rpx），或按钮底色微微变化（如从 `#252540` 变为 `rgba(139,92,246,0.08)`） |

### 2.2 重复上次

| | 现有设计 | 调整方案 |
|---|---|---|
| **布局** | 左侧："上次: 早起 (生活习惯)"，右侧："重复 →"。整行用 `flex-between`，bindtap 绑定在整行 | 视觉上"重复 →"作为文字链接不够醒目，且用户可能不知道整行可点击。改为：左侧保持文字信息不变，右侧"重复"改为带边框的小胶囊按钮（和"炼化规则"按钮一致的样式），整行底部分割线保留 |

### 2.3 分类选择流程

| | 现有设计 | 调整方案 |
|---|---|---|
| **步骤层级** | 快捷打卡卡片在上，"选择行为类型"卡片紧跟在下，两者视觉权重相同 | 两个区域之间加一个**分割提示**：一行灰色文字"── 或选择其他行为 ──"，明确传达"上面是快捷方式，下面是完整入口"的主次关系 |
| **分类按钮** | `flex-wrap` 排列，按钮之间 gap 16rpx，每个按钮 `padding: 20rpx 32rpx`，按钮内容为纯文字（如"身体健康""学习""生活习惯"） | 每个分类按钮左侧增加一个对应属性的色点（8rpx 圆形），颜色与属性颜色一致（身体健康 = 体魄红 `#e57373`，学习 = 悟性蓝 `#64b5f6`，生活习惯 = 心性紫 `#ba68c8`，家务 = 灵巧绿 `#81c784`，社交互助 = 神识橙 `#ffb74d`），让用户在选分类时就能预知"做这类行为会提升什么属性" |

### 2.4 打卡成功反馈（系统设计 + 交互设计）

| | 现有设计 | 调整方案 |
|---|---|---|
| **反馈方式** | `wx.showToast({ title: '获得: 造化工巧丹(凡品)', icon: 'success', duration: 2000 })`——2 秒后自动消失，只展示道具名和品质 | 改用 `wx.showModal` 或自定义弹窗，内容分三层信息：**第一层（即时收益）**："获得 造化工巧丹（凡品）· 灵巧 +1 临时属性"；**第二层（积累进度）**："灵巧道具累计 7/10 点"；**第三层（下一步引导）**：若累计 ≥ 10，显示"可以去背包炼化了！"按钮跳转背包页；若 < 10，显示"继续加油，还差 N 点"。确认按钮关闭弹窗。这是整个产品体验中**因果感知断裂**的核心修复点 |
| **数据来源** | 当前 API `POST /behavior` 的返回值 `res` 包含 `item`（道具信息）和 `cultivationStatus`，但没有该属性维度的累计临时值 | 后端 API 需在返回值中增加 `attrTempTotal` 字段：当次行为对应的属性维度当前累计临时值（炼化前的总值），前端用于展示进度 |

## 三、背包页

### 3.1 道具品质的视觉层次

| | 现有设计 | 调整方案 |
|---|---|---|
| **品质区分** | 品质仅通过文字颜色区分：凡品灰 `#94a3b8`、良品绿 `#10b981`、上品蓝 `#3b82f6`、极品金 `#f59e0b`。道具卡片本身的背景、边框对所有品质一致 | 道具行（`.item-row`）左侧增加一条 4rpx 宽的**品质色竖条**（通过 `border-left` 实现）。凡品无竖条（保持现状），良品绿条，上品蓝条，极品金条。让用户在扫视列表时立刻识别品质分布 |
| **品质标签** | 道具名右侧有一个小标签显示品质文字（如"凡品"），背景为 `rgba(255,255,255,0.06)`，非常淡 | 标签底色跟随品质色：凡品 `rgba(148,163,184,0.12)`、良品 `rgba(16,185,129,0.15)`、上品 `rgba(59,130,246,0.15)`、极品 `rgba(245,158,11,0.15)`，让标签本身就有品质感 |

### 3.2 炼化操作引导

| | 现有设计 | 调整方案 |
|---|---|---|
| **炼化规则入口** | 右上角紫色胶囊按钮"炼化规则"，带边框和半透明背景 | 位置和样式保持不变（已经比较合理），但**首次进入背包页**时自动弹出炼化规则弹窗（通过 localStorage 记录是否首次），让新用户第一时间理解玩法 |
| **未选中道具时** | 底部合成栏完全隐藏，用户看不到任何操作提示 | 底部固定展示一行提示栏："选择同属性道具 → 累计 10 点即可炼化"，用灰色文字 + 较低视觉权重，点击后展开炼化规则。选中道具后此提示栏被合成摘要栏替换 |
| **合成摘要栏** | 选中道具后底部出现："已选 N 件 · 总值 X" + "可合成 +Y 永久属性"或"不足10点无法合成" + 合成按钮 | 增加一个**进度条可视化**：在文字上方加一条横向进度条，填充色为紫色，以 10 的倍数为刻度。如总值 7 时进度条填到 70%，总值 13 时填满一格（10）+ 第二格填到 30%。让"凑整"的概念更直观 |

### 3.3 属性分类标签

| | 现有设计 | 调整方案 |
|---|---|---|
| **未选中态** | `background: #252540; color: #94a3b8; border: none` | 保持现状，但在标签文字前增加对应属性的色点（和行为页分类按钮方案一致），强化属性 → 颜色的认知映射 |
| **选中态** | `background: #8b5cf6; color: #fff` | 保持现状 |
| **滚动** | 使用 `<scroll-view scroll-x>`，当分类超过一屏时可横向滚动 | 保持现状。但如果分类只有 3-4 个（不超过屏宽），去掉 scroll-view 改为 flex 横排，避免横向滚动的暗示（用户可能不知道能滚） |

## 五、家庭页

### 5.1 成员卡片信息重构

| | 现有设计 | 调整方案 |
|---|---|---|
| **头像** | 80rpx 圆形，背景色为高饱和度色（如亮黄 `#f59e0b` 系），内显首字母大写 | 头像缩小至 **64rpx**，背景色饱和度降低（在现有色值基础上混入 20% 暗底色，或 opacity: 0.75），让头像不再是视觉焦点 |
| **状态小绿点** | 20rpx 圆点，border 4rpx solid #1a1a2e，在头像右下角 | 增大至 **24rpx**，border 改为白色 `#f8fafc` 以增强可见度。在线为绿色，离线为灰色 `#475569` |
| **卡片内容** | 只显示：头像 + 名字 + 境界徽章。所有人显示相同的"练气一阶"，大量视觉重复 | 增加一行**差异化信息**：显示"本周 N 次修炼"或"今日已修炼 ✓"/ "今日未修炼"。数据来源：后端 family API 在每个成员对象中增加 `weeklyBehaviorCount` 和 `todayActive` 字段 |
| **境界徽章** | 渐变色胶囊，所有人相同时造成视觉噪音 | 当全家境界相同时，徽章改为**简化显示**：灰色小字替代渐变胶囊（如"练气一阶"用 `font-size: 20rpx; color: #64748b` 显示），不再使用渐变背景。只有境界不同的成员才用渐变徽章，突出差异 |

### 5.2 动态流优化

| | 现有设计 | 调整方案 |
|---|---|---|
| **动态文案** | "完成了 洗碗"+ 品质标签"凡品"+ "获得 乾坤挪移诀" | 调整语序：**"完成了 洗碗 · 获得 乾坤挪移诀"**，品质标签跟在道具名后面而不是行为名后面（品质描述的是道具而非行为）。品质标签样式保持现有 `.quality-tag` |
| **头像尺寸** | 动态流中头像 72rpx | 保持 72rpx 不变（动态流中头像作为内容锚点需要一定尺寸） |

### 5.3 反应按钮

| | 现有设计 | 调整方案 |
|---|---|---|
| **按钮尺寸** | `min-height: 72rpx; padding: 12rpx 16rpx`，实际触摸区域约 36px 高度，低于 44px 推荐最小触摸目标 | 调整为 `min-height: 80rpx; padding: 16rpx 20rpx`，确保触摸区域 ≥ 44px |
| **反馈机制** | 点击 emoji 后发送 reaction，按钮高亮（`.reaction-active` 加紫色边框），但用户不知道被赞者是否会收到通知或获得收益 | **交互层面**：点击后增加一个轻量 toast 提示"已为 [用户名] 点赞"，让操作有确认感。**系统设计层面（可选）**：每条动态的首个点赞为被赞者增加 0.1 临时属性值（取被赞行为对应的属性维度）。在 toast 中提示"为 [用户名] 增加了 0.1 [属性名] 临时值"。此机制需后端支持，让社交互动有实际游戏意义 |

### 5.4 家庭竞争/协作机制（新增模块建议）

| | 现有设计 | 调整方案 |
|---|---|---|
| **无排行/对比** | 家庭页仅展示成员列表和动态流，无任何对比或竞争元素 | 在成员卡片和动态流之间，新增一个**"本周修炼榜"卡片**。不用数字排名（避免伤害积极性），而是用**横向进度条对比**：每个成员一行，左侧名字，右侧进度条（长度 = 本周行为次数 / 最多者次数 × 100%），颜色用各自头像色。表头显示"本周家庭共修炼 N 次"。此功能需后端 family API 增加 `weeklyLeaderboard` 数据 |

## 六、全局交互与系统设计

### 6.1 跨页面因果链（系统设计）

| 环节 | 现有设计 | 调整方案 |
|---|---|---|
| **行为 → 道具** | 打卡成功后 toast 显示道具名，但不告诉用户该道具属于什么属性、累计了多少 | 见行为页 2.4 节方案 |
| **道具 → 属性** | 背包页炼化成功后跳转或提示不明确 | 炼化成功后弹出 Modal："[属性名] 永久提升 +N！当前 [属性名] = X"，确认后自动刷新首页雷达图数据 |
| **属性 → 愿望** | 愿望页"Boss 战力 18~22"但用户不知道自己当前战力 | 见愿望页 4.1 节方案 |
| **愿望 → 奖励** | 挑战成功后有胜利卡片，但奖励兑现在背包页的"奖励"Tab 中，用户可能找不到 | 胜利卡片底部增加按钮"去兑现奖励 →"，点击跳转到背包页并自动切换到"奖励"Tab |

### 6.2 按钮触摸反馈（交互设计）

| | 现有设计 | 调整方案 |
|---|---|---|
| **:active 反馈** | 各按钮使用 `opacity: 0.7` 或 `opacity: 0.8` 作为点击态，反馈时间极短（手指松开即恢复），用户难以感知 | 统一所有可交互元素的 active 态为 `opacity: 0.85; transform: scale(0.97)`，增加"按下去"的物理感。对于关键操作按钮（如"合成""挑战""提交打卡"），再叠加背景色加深效果 |

### 6.3 空状态设计（UI 设计）

| 页面 | 现有空状态 | 调整方案 |
|---|---|---|
| **背包** | "背包空空如也" + "去上报行为"按钮 | 保持按钮，文案改为引导性的："完成行为即可获得道具，快去修炼吧" |
| **愿望** | 🌟 + "还没有愿望，许一个试试？" | 保持现状，已经合理 |
| **家庭动态** | "暂无动态" | 改为"家庭成员的修炼动态会出现在这里" |
| **首页属性全0** | 照常渲染最小雷达图 | 见首页 1.3 节方案 |

### 6.4 数据加载状态（交互设计）

| | 现有设计 | 调整方案 |
|---|---|---|
| **加载中** | 各页面独立的"加载中..."纯文字，无动画 | 统一为骨架屏（skeleton screen）：用灰色块占位卡片、标题、进度条的位置，页面加载完成后骨架屏渐隐切换为真实内容。减少"空白跳动"的感觉 |
| **首页首次加载** | `wx:if="{{!loading}}"` 控制整个页面容器，加载时只显示居中文字 | 首页角色卡片和推荐卡片优先渲染（这两个是高优先级信息），趋势图和成就可以延迟加载（低优先级信息），避免用户等待全部数据 |

### 6.5 Tab Bar 图标（UI 设计）

| | 现有设计 | 调整方案 |
|---|---|---|
| **底部 Tab** | 只有文字（"角色""行为""背包""愿望""家庭"），无图标 | 为每个 Tab 增加图标（可用简单线性图标 PNG）：角色/行为/背包/愿望/家庭 各一对（未选中 + 选中态）。在 `app.json` 的 tabBar.list 中配置 `iconPath` 和 `selectedIconPath`。纯文字 Tab 在 5 个选项时辨识度偏低 |

## 七、新增功能建议

以下为当前版本未涉及但对用户留存和体验有显著价值的功能建议，按可行性和影响排序。

### 7.1 每日修炼摘要推送（系统设计）

| 维度 | 说明 |
|---|---|
| **问题** | 用户如果今天忘了打卡，没有任何提醒机制。衰退系统是负面惩罚，但用户感知不到"正在衰退" |
| **方案** | 利用微信小程序的**订阅消息**能力，每天晚上 20:00 推送一条摘要："今日修炼 N 次，[属性名] +X 临时值。距离炼化还差 Y 点。" 未修炼时推送："今日尚未修炼，连续修炼 Z 天可达到精进状态。" |
| **实现依赖** | 需用户在小程序内授权订阅消息模板，后端增加定时任务 |
| **优先级** | P2——对留存有直接帮助，但需要微信消息模板审核 |

### 7.2 衰退预警可视化（系统设计 + UI 设计）

| 维度 | 说明 |
|---|---|
| **问题** | PRD 中设计了衰退机制（15 天缓冲 → 每日衰退），但当前 UI 上**无任何衰退预警展示**。用户不知道哪个属性即将衰退 |
| **方案** | 首页属性总览中，对即将进入衰退期（缓冲期剩余 ≤ 3 天）的属性增加**红色预警标记**。雷达图中该属性的顶点增加闪烁动画。属性标签旁显示"⚠ 3 天后开始衰退"。点击可跳转到对应行为分类的打卡页 |
| **数据来源** | 后端 `GET /character` 返回值中增加 `decayWarnings` 数组：`[{ attr: 'physique', daysUntilDecay: 3 }]` |
| **优先级** | P1——衰退是核心系统机制，没有可视化等于不存在 |

### 7.3 境界突破仪式（交互设计）

| 维度 | 说明 |
|---|---|
| **问题** | PRD 中设计了境界系统（练气一阶到十阶），但当前属性总和达到升阶条件时，用户感知不到境界变化的时刻 |
| **方案** | 当属性总和达到下一阶要求时，首页角色卡顶部出现**突破提示横幅**："修为已足，可突破至练气二阶"。点击后进入一个简单的**突破动画页面**（全屏渐变 + 境界文字渐显 + 意境描述），确认后完成突破。将升阶从"静默计算"变为"用户主动触发的仪式" |
| **实现** | 后端增加 `canBreakthrough` 字段和 `POST /character/breakthrough` 接口。前端增加突破动画页（可做为 home 页内弹窗） |
| **优先级** | P2——增强成就感和里程碑体验 |

### 7.4 行为连续天数可视化（UI 设计）

| 维度 | 说明 |
|---|---|
| **问题** | PRD 中打卡型行为的品质由连续天数决定（3天良品/7天上品/14天极品），但用户在行为页看不到当前连续天数 |
| **方案** | 快捷打卡按钮上增加**连续天数角标**。如"早起"按钮右上角显示"🔥7"表示连续 7 天。行为表单页也显示："当前连续 7 天 → 本次品质：上品" |
| **数据来源** | 后端 shortcuts 和 behavior 返回值中增加 `streak` 字段 |
| **优先级** | P1——直接影响用户对品质判定的理解和打卡动力 |

### 7.5 道具图鉴 / 收集进度（系统设计）

| 维度 | 说明 |
|---|---|
| **问题** | 道具名称有丰富的修仙主题设计（铁骨丹、淬体散、筋脉强化符等），但获得后炼化就消失了，用户对道具没有收集感 |
| **方案** | 新增**道具图鉴页**（可作为背包页的第三个 Tab）。记录用户历史上获得过的所有道具种类，按属性分类展示。未获得的道具显示为灰色轮廓 + "???"。图鉴完成度 = 已获得种类 / 总种类数 |
| **数据来源** | 后端新增道具图鉴表，记录每个用户的道具获取历史（去重） |
| **优先级** | P3——增加收集乐趣，但不影响核心循环 |

### 7.6 家庭周报（系统设计 + UI 设计）

| 维度 | 说明 |
|---|---|
| **问题** | 行为页的"历史"Tab 有个人周报，但没有**家庭维度的周期性复盘** |
| **方案** | 每周一生成一份**家庭周报卡片**，展示在家庭页顶部（可折叠）。内容包括：(1) 全家本周总行为次数 + 和上周对比；(2) 最勤奋成员（本周行为最多者）；(3) 全家道具获取分布（品质饼图）；(4) 团队愿望进度更新。可增加一句趣味化总结："本周全家共修炼 47 次，灵气充盈，道心稳固。" |
| **实现** | 后端增加周报数据聚合接口 `GET /family/weekly-report`，前端在家庭页条件渲染 |
| **优先级** | P2——增强家庭协作感和复盘习惯 |

### 7.7 快捷打卡 Widget / 桌面入口（交互设计）

| 维度 | 说明 |
|---|---|
| **问题** | 每次打卡都需要：打开微信 → 进入小程序 → 切换到行为页 → 选择分类 → 选择行为 → 提交。高频操作路径太长 |
| **方案** | (1) **小程序快捷方式**：利用微信「添加到桌面」能力，引导用户将小程序添加到手机桌面，减少打开路径；(2) **场景值跳转**：通过 `pages/behavior/behavior?action=quick&subtype=早起` 形式的页面参数，支持从微信对话中直接跳转到指定行为的打卡确认（可配合订阅消息中的跳转链接）；(3) **快捷打卡浮窗**：行为页增加一个浮动按钮（FAB），点击后弹出快捷打卡列表（仅展示 top 3 常用行为），一键提交 |
| **优先级** | P2——降低核心操作的摩擦 |

### 7.8 属性衰退恢复通知（系统设计）

| 维度 | 说明 |
|---|---|
| **问题** | 用户做了一次行为后重置了衰退计时器，但没有任何正面反馈告知"衰退已停止" |
| **方案** | 当某属性处于衰退期或临近衰退时，用户完成对应行为后，打卡成功弹窗中增加一行**恢复提示**："✓ [属性名] 衰退已重置，15 天缓冲期重新开始"。让用户感受到"我的行为阻止了属性流失"这一正面因果 |
| **优先级** | P1——低成本、高感知的因果反馈 |

### 7.9 新手引导流程（交互设计）

| 维度 | 说明 |
|---|---|
| **问题** | 新用户登录后直接进入首页，面对空白的雷达图、空的趋势图、全 0 的属性，不知道该做什么。"今日推荐"卡片虽然有引导，但视觉层级不够突出 |
| **方案** | 首次登录用户增加**3 步新手引导**（蒙层 + 高亮提示）：① "这是你的修仙角色，先去记录第一个行为吧" → 高亮底部 Tab "行为"；② 进入行为页后 → "选择一个行为类型开始修炼" → 高亮分类按钮区域；③ 打卡成功后 → "恭喜获得第一个道具！积累道具后可在背包炼化为永久属性" → 引导查看背包。引导完成后设置 localStorage 标记，不再重复 |
| **优先级** | P2——对新用户转化率有直接帮助 |

### 7.10 成就系统扩展（系统设计）

| 维度 | 说明 |
|---|---|
| **问题** | 当前成就系统已有框架（首页可折叠的成就列表），但从截图看成就和日常行为的关联不够紧密 |
| **方案** | 增加与核心循环强相关的成就条目：(1) **行为维度**：首次打卡、连续 7 天、连续 30 天、累计 100 次、5 类行为都做过；(2) **道具维度**：首次获得良品/上品/极品、首次炼化、炼化 10 次；(3) **愿望维度**：首次挑战、首次胜利、连胜 3 次、团队愿望全员通过；(4) **社交维度**：首次给家人点赞、收到 10 个赞。每个成就解锁时首页弹出成就卡片动画 |
| **优先级** | P3——长期留存机制 |

## 十、评审决策记录（2026-04-20）

### 采纳项（按执行顺序）
1. P0-1 打卡反馈重构：toast→Modal，三层信息（即时收益→积累进度→下一步引导），后端加 attrTempTotal
2. P1-3 streak 连续天数展示：快捷按钮角标+表单页显示（配合策划案-01）
3. P1-5 + P1-6 背包页品质色竖条 + 炼化引导（纯前端）
4. P0-2 修炼进度引导：改为弹窗形式（非固定卡片），不占首页空间
5. P2-2 跨页面因果链：炼化成功 Modal、胜利后"去兑现奖励"按钮

### 调整项
- 境界意境描述：暂不做固定文案，等修炼日志系统做好后再融入首页
- 修炼进度引导：从独立卡片改为弹窗引导
- 趋势图：不折叠，改为最右侧固定今天、往左倒序排列7天

### 暂缓项
- 骨架屏、订阅消息推送、新手引导蒙层（家庭内部产品投入产出比低）
- 本周修炼榜：记录待议，如果做用合作视角（"全家共修炼N次"）而非排名
