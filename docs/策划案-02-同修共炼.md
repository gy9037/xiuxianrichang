# 策划案-02：同修共炼（多人行为）

> **状态**：待评审
> **优先级**：P2
> **关联系统**：行为系统、道具系统、家庭系统
> **新增文件**：无
> **修改文件**：`server/routes/behavior.js`、`server/db.js`（新增表）

---

## 一、背景与设计目标

### 现状

当前行为系统是纯个人行为——每条记录都属于单一用户，家庭成员之间的真实共同活动（一起打球、一起桌游、一起做饭）在系统里只能各自独立打卡，彼此没有关联。

这意味着：
- 一家三口周末一起运动，从系统视角看是三个毫无关系的独立行为
- 家庭共同活动得不到任何额外认可，家庭互动的激励价值被忽略

### 设计目标

在行为系统中增加**共修类**行为，支持多人参与同一条行为记录。当家庭成员真实地在一起做某件事，这件事应该作为一个整体被记录下来，并对所有参与者产生有意义的反馈。

### 设计原则

- **真实性优先**：需要每位参与者主动确认，防止代打卡
- **行为自然延伸**：共修类是现有五类行为之外的第六类，不替代个人行为
- **轻量操作**：发起和确认都是极少步骤，不打断日常流程

## 二、共修类行为设计

### 2.1 行为类别定义

新增第六个行为类别：**共修类**，专门用于记录多人共同完成的活动。

| 属性 | 说明 |
|------|------|
| 类别名 | 共修类 |
| 对应属性 | 由发起时选择的**主题**决定（见下表） |
| 参与人数 | 最少 2 人，上限为家庭总人数 |
| 道具产出 | 每位参与者各产一件，品质有共修加成 |

### 2.2 共修主题与属性对应

发起共修时，需选择一个主题方向，决定本次活动产出的道具属性：

| 主题 | 示例活动 | 对应属性 |
|------|---------|---------|
| 强健体魄 | 一起跑步、打球、骑行、爬山 | 体魄 |
| 增益悟性 | 一起读书讨论、看纪录片、下棋 | 悟性 |
| 修养心性 | 一起做家务、整理房间、静坐 | 心性 |
| 磨砺灵巧 | 一起桌游、拼图、手工制作、麻将 | 灵巧 |
| 感知相连 | 一起聊天散步、家庭聚餐、陪伴 | 神识 |

### 2.3 共修加成规则

多人共同参与，道具品质判定有额外加成（叠加在 streak 品质基础上）：

| 确认参与人数 | 品质加成 |
|------------|---------|
| 2 人 | 良品概率 +10% |
| 3 人 | 良品概率 +20%，上品概率 +5% |
| 全家（≥4 人） | 良品概率 +30%，上品概率 +10% |

> 加成叠加计算，不替换原有 streak 品质判定逻辑。

## 三、操作流程

### 3.1 发起共修

1. 发起者进入行为页，点击「发起共修」（行为页新增入口）
2. 填写：活动名称（可选，默认使用主题名）、选择主题、选择邀请的家庭成员
3. 提交后，被邀请成员收到通知

### 3.2 确认参与

- 被邀请成员在**行为页顶部**看到「共修邀请」提示
- 点击查看详情（发起人、主题、发起时间）
- 点击「确认参与」或「不参与」
- **24 小时内未响应，视为不参与**

### 3.3 行为生成

- 发起者确认提交时，所有已确认参与的成员各自生成一条共修类行为记录
- 发起者不需要额外操作，提交即完成
- 每位参与者的行为记录独立存储（`user_id` 不同），但通过 `group_id` 关联

### 3.4 奖励发放

- 每位参与者各自产出一件道具，属性对应共修主题
- 品质判定：该用户当前该属性类别的 streak 基础 + 共修人数加成
- 未确认参与的成员不产出道具

## 四、展示设计

### 4.1 家庭动态页

全家共修时，家庭动态发出特殊公告：

> 🌟「全家共修」今日云鹤居士发起强健体魄共修，全家四人参与，各获体魄道具一件。

### 4.2 修炼日志联动

当日有共修记录时，修炼日志插入专属句：

> 今日与家人同修{主题}，道气相合，情谊更深。

### 4.3 行为列表显示

共修行为在行为记录列表中显示特殊标记：

```
[共修] 强健体魄   今日 · 全家 4 人参与   →  体魄道具（良品）
```

## 五、数据库设计

### 新增表：`group_behaviors`（共修事件表）

```sql
CREATE TABLE group_behaviors (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  initiator_id INTEGER NOT NULL,
  family_id INTEGER NOT NULL,
  theme TEXT NOT NULL,           -- '体魄'|'悟性'|'心性'|'灵巧'|'神识'
  activity_name TEXT,            -- 可选，活动名称
  status TEXT DEFAULT 'pending', -- 'pending'|'completed'|'expired'
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  expires_at DATETIME NOT NULL   -- created_at + 24小时
);
```

### 新增表：`group_behavior_participants`（参与确认表）

```sql
CREATE TABLE group_behavior_participants (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  group_behavior_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  status TEXT DEFAULT 'invited',  -- 'invited'|'confirmed'|'declined'|'expired'
  confirmed_at DATETIME,
  behavior_id INTEGER,            -- 生成的行为记录 ID（确认后填入）
  UNIQUE(group_behavior_id, user_id)
);
```

### behaviors 表无需改动

共修行为在 `behaviors` 表中正常存储，`category` 为 `'共修'`，通过 `group_behavior_id` 外键关联。

## 六、后端改动清单

| 新增/修改 | 说明 |
|---------|------|
| `POST /api/group-behavior` | 发起共修，创建事件 + 发送家庭内通知 |
| `POST /api/group-behavior/:id/confirm` | 参与者确认参与 |
| `POST /api/group-behavior/:id/decline` | 参与者拒绝 |
| `GET /api/group-behavior/pending` | 获取当前用户待确认的共修邀请 |
| `server/routes/behavior.js` | 共修行为提交时批量生成各参与者的行为记录，计算共修加成 |

## 七、待确认问题

1. 共修行为是否也算入各参与者对应属性类别的 streak？建议：算入，这样共同活动同时推进个人 streak，形成正向激励
2. 发起者自己是否默认参与？建议：默认勾选发起者，可取消
3. 活动名称是否必填？建议：非必填，不填时展示"[主题名]共修"
