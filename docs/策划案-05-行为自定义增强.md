# 策划案-05：行为自定义增强

> **状态**：待评审
> **优先级**：P2
> **关联系统**：行为系统、修炼日志
> **修改文件**：`server/routes/behavior.js`、`server/routes/character.js`、miniprogram 行为页

---

## 一、背景与设计目标

当前行为页的快捷打卡按钮是系统自动排序的（按历史使用频次），用户没有任何控制权。打卡时也没有任何个性化空间——每次提交都是一样的流程，和打卡内容完全脱节。

本策划案包含三个独立增强，可分批上线：

| 功能 | 核心价值 |
|------|---------|
| 快捷按钮置顶 | 让用户对最重要的习惯有控制感 |
| 行为备注 | 让打卡有个性化记忆，联动修炼日志 |
| 月度目标 | 给用户一个清晰的方向感，首页可见 |

---

## 二、功能一：快捷打卡按钮手动置顶

### 2.1 现有设计

快捷打卡按钮区展示用户最常使用的 5 个 sub_type，排序完全由后端按使用频次决定，用户无法调整。

### 2.2 调整方案

**允许用户手动置顶 1-2 个常用行为**，固定出现在快捷按钮区的前两位，其余位置仍按频次自动排序。

**操作方式**：

- 长按快捷按钮区中的任意按钮 → 弹出操作菜单：「置顶显示」/「取消置顶」
- 或在行为页设置入口（右上角）进行管理
- 最多同时置顶 2 个，超出时提示"已达置顶上限，请先取消一个"

**数据存储**：

在 `characters` 表或新增 `user_preferences` 表中增加字段 `pinned_behaviors`，存储 JSON 数组（最多 2 条 sub_type）：

```json
["早起", "跑步"]
```

**展示规则**：

```
[置顶] 早起  ★  |  [置顶] 跑步  ★  |  读书  |  家务  |  冥想
```

置顶的按钮右上角显示小星标，与普通按钮有视觉区分。

## 三、功能二：行为备注

### 3.1 现有设计

打卡提交时没有任何自由输入项，系统只记录 category、sub_type、intensity（如有）等结构化字段。

### 3.2 调整方案

**允许用户在打卡时输入一行短备注**（≤ 20 字），记录当次打卡的具体内容或感受。

**UI 设计**：

```
[行为提交弹窗]
─────────────────────────────────
  早起
  ─────────────────────────────
  今天的状态怎么样？（可选）
  ┌──────────────────────────┐
  │ 早起跑了步，天气很好      │
  └──────────────────────────┘
                          4/20
─────────────────────────────────
          [ 完成打卡 ]
```

- 输入框折叠默认隐藏，点击"今天的状态怎么样？"展开
- 不填写直接提交也可以，备注为可选项
- 字数限制 20 字，超出时截断

### 3.3 备注的使用场景

**① 修炼日志联动**

有备注时，日志中的行为描述句使用备注内容替代默认模板：

- 无备注：`"身体力行，晨跑锻炼，筋骨得以淬炼。"`
- 有备注（"早起跑了步，天气很好"）：`"今日早起，早起跑了步，天气很好，筋骨得以淬炼。"`

**② 行为历史记录**

在行为记录列表中，有备注的行为展示一行备注文字，让历史记录更有回忆价值。

### 3.4 数据存储

`behaviors` 表增加 `note` 字段：

```sql
ALTER TABLE behaviors ADD COLUMN note TEXT;
```

## 四、功能三：月度目标设定

### 4.1 现有设计

用户没有目标设定功能，首页也没有任何进度展示，用户对"本月要做什么"没有清晰预期。

### 4.2 调整方案

**允许用户为任意 sub_type 设定本月目标次数**，在角色页或行为页展示当前完成进度。

**设定入口**：

行为页 → 点击某个 sub_type 的行为 → 详情页底部「设定本月目标」

**目标格式**：

「本月完成 [sub_type] × N 次」，N 由用户输入（建议范围 3-30 次）。

**展示位置**：

在角色页「今日概况」下方新增「本月目标」区块（仅显示已设定目标的 sub_type）：

```
┌────────────────────────────────────┐
│ 本月目标                            │
│                                    │
│ 早起    ▓▓▓▓▓▓▓░░░  14/20 次       │
│ 跑步    ▓▓▓░░░░░░░   6/15 次       │
│ 读书    ▓▓▓▓▓░░░░░  10/20 次       │
│                         [管理目标] │
└────────────────────────────────────┘
```

**目标达成通知**：

目标达成时（完成 N 次），触发一次修炼弹窗提示：

> 「早起」本月目标达成！已完成 20/20 次，道心坚定。

### 4.3 数据存储

新增表 `behavior_goals`：

```sql
CREATE TABLE behavior_goals (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  sub_type TEXT NOT NULL,
  target_count INTEGER NOT NULL,
  period_key TEXT NOT NULL,       -- 格式：'2026-04'（年月）
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(user_id, sub_type, period_key)
);
```

目标进度实时从 `behaviors` 表查询（`COUNT(*)` where `user_id + sub_type + 当月`），不需要额外存储进度值。

## 五、后端改动清单

| 修改文件 | 改动 |
|---------|------|
| `server/routes/behavior.js` | `POST /behavior` 接受并存储 `note` 字段 |
| `server/routes/character.js` | `GET /character` 返回 `pinned_behaviors` 字段 |
| `server/routes/character.js` | `PATCH /character/pin-behavior` 更新置顶行为配置 |

| 新增路由 | 说明 |
|---------|------|
| `POST /api/behavior-goal` | 创建/更新某 sub_type 的月度目标 |
| `GET /api/behavior-goal/current` | 获取当月所有目标及进度 |
| `DELETE /api/behavior-goal/:id` | 删除目标 |

## 六、上线优先级建议

三个功能相互独立，建议按以下顺序分批上线：

1. **行为备注**（最简单，修炼日志上线时同步）
2. **快捷按钮置顶**（独立上线，用户控制感强）
3. **月度目标**（配合修炼日志一起上线效果最好）
