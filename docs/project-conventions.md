# 项目工作规范

> 创建日期：2026-04-09
> 适用范围：修仙日常项目全生命周期
> 维护者：主 Agent（每次 session 启动时读取本文档）

---

## 一、角色分工

项目由主 Agent 主导推进，根据任务性质分配不同角色处理。

| 角色 | 职责 | 执行方 |
|------|------|--------|
| 主 Agent | 项目管理、策划案编写、技术方案设计、任务分配、文档维护、质量把控 | AI（当前 session） |
| 交互体验优化师 | UX 审查、体验问题梳理、改进建议 | 子 Agent |
| 其他专项角色 | 按需分配（如数据分析、文案设计等） | 子 Agent |
| Codex 执行 | 代码开发、bug 修复 | 用户通过 Codex 执行 |
| 决策确认 | 方案审批、优先级确认、验收 | 用户（Gavin） |

关键原则：
- 策划案、技术方案、审查报告等文档类工作由主 Agent 或子 Agent 完成，落实到文件
- 代码开发输出 Codex 指令文档，由用户执行
- 子 Agent 任务要控制粒度，避免超时（参见 `rules/skills/workflow_subagent_guidelines.md`）

## 三、文档体系

### 目录结构

所有项目文档存放在 `docs/` 下：

| 文件 | 用途 |
|------|------|
| `PRD.md` | 产品需求文档（原始需求） |
| `discussion-log.md` | 讨论记录与决策日志 |
| `tech-architecture.md` | 技术架构方案 |
| `project-conventions.md` | 本文档（工作规范） |
| `iteration-vX.md` | 版本策划案 |
| `tech-vX-fXX.md` | 功能技术方案 |
| `codex-vX-fXX.md` | Codex 执行指令 |
| `ux-review-{page}.md` | UX 审查报告 |

### 策划案格式

每个版本策划案（`iteration-vX.md`）包含：
1. 需求来源索引（列出所有原始来源及编号）
2. 版本方向说明
3. 按优先级分组的功能清单（P0/P1/P2）
4. 每条标注：编号、问题描述、来源、严重度、修复方向
5. 版本边界说明（明确不做什么）

## 四、工作流程

### 新版本策划流程

1. 收集输入：UX 审查报告 + 用户反馈 + 遗留问题
2. 交叉比对去重，统一编号
3. 按严重度和影响范围分 P0/P1/P2
4. 每条关联来源编号
5. 输出策划案，提交用户确认

### 开发执行流程

1. 策划案确认后，按优先级编写技术方案
2. 技术方案关联策划编号
3. 输出 Codex 指令文档，关联技术方案
4. 用户执行 Codex 指令
5. 验收：对照策划条目逐条检查
6. 通过后 git commit，commit message 关联版本号

### 子 Agent 使用规范

参见 `rules/skills/workflow_subagent_guidelines.md`，核心要点：
- 每个子 Agent 任务控制在可完成范围内
- 合并类任务（多文件汇总）由主 Agent 直接处理
- 派出后等系统通知，不轮询

## 六、任务完成报告规范

当用户说"输出完成报告"时，执行以下操作：

1. 读取 `docs/v127-task-tracker.md`
2. 将对应任务状态改为 `已完成`（或 `部分完成` / `有阻塞`）
3. 在文件末尾"完成摘要"章节下追加报告，格式：

```
### [任务名]（[日期]）

- 状态：已完成 / 部分完成 / 有阻塞
- 改动文件：[修改的文件路径，每行一个]
- 关键决策：[偏离原计划的设计决策，没有写"无"]
- 遗留问题：[未解决的问题，没有写"无"]
```

约束：整段报告 10 行以内，不展开技术细节，只记录结论。
