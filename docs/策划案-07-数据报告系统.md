## 一、背景与设计目标

修炼日志（策划案-08）提供了日/周/月三个层级的自动叙事。数据报告系统是更上层的**阶段性总结**，以月、季、年为单位，提炼用户在较长周期内的成长轨迹，并生成**可分享的图卡**。

两者的定位区别：

| 维度 | 修炼日志 | 数据报告 |
|------|---------|---------|
| 周期 | 日 / 周 / 月 | 月 / 季 / 年 |
| 内容 | 叙事性文字，有故事感 | 数据提炼 + 成就感 |
| 形式 | 文字为主 | 图卡为主，可保存分享 |
| 生成方式 | 懒生成（用户访问时触发） | 懒生成（用户访问时触发） |

---

## 二、报告类型与触发时机

| 报告类型 | 生成时间 | 覆盖周期 |
|---------|---------|---------|
| **月报** | 用户访问时实时生成并缓存 | 上一个自然月（如 4 月访问时生成 3 月报告） |
| **季报** | 用户访问时实时生成并缓存 | 上一个季度 |
| **年报** | 用户访问时实时生成并缓存 | 上一个自然年 |

> 三种报告均在用户首次访问时实时生成，生成后缓存到 reports 表，后续访问直接读取缓存。
> 覆盖周期内无任何行为记录的用户，不生成报告。

## 三、月报内容设计

### 3.1 内容结构

| 板块 | 内容 | 版本 |
|------|------|------|
| 修炼天数 | 当月有行为记录的天数（如：23天） | 第一版 |
| 活跃度评级 | 按天数比例给出等级（见下表） | 第一版 |
| 五属性成长 | 各属性本月临时属性积累量（可炼化的净增值） | 第一版 |
| 最长 Streak | 本月坚持最久的 sub_type 及连续天数 | 第一版 |
| 道具收获 | 本月获得的道具总数，各品质数量分布 | 第一版 |
| 节气经历 | 本月内经过的节气名称，获得了几种限定道具 | 后续扩展（节气系统上线后） |
| 境界变化 | 本月是否有境界突破（有则展示突破前后） | 第一版 |

### 3.2 活跃度评级

| 活跃天数 / 当月天数 | 评级 | 修仙文案 |
|------------------|------|---------|
| < 30% | 散漫 | 修炼时断时续，道心尚需磨砺 |
| 30%-59% | 勤勉 | 已见道心萌动，继续坚持 |
| 60%-89% | 精进 | 修炼颇为稳定，灵台清明 |
| ≥ 90% | 至诚 | 月无虚日，道心坚韧堪称楷模 |

## 六、图卡生成技术方案

### 6.1 生成方式

使用**微信小程序 Canvas API**在客户端绘制图卡：

- 后端提供报告数据（JSON）
- 前端 Canvas 绘制图卡内容
- 用户可保存至相册（需授权）或直接分享到微信

### 6.2 图卡尺寸

推荐 750 × 1000 px（微信朋友圈竖版最佳比例）。

### 6.3 字体与风格

- 背景：深色系（墨黑 `#1a1a2e` 或深紫 `#16213e`）
- 强调色：金色 `#f59e0b`（标题、数字）
- 正文：白色 `#f1f5f9`
- 整体风格：水墨风修仙，与 App 主色调一致

## 七、数据库设计

### 新增表：`reports`

```sql
CREATE TABLE reports (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  type TEXT NOT NULL,         -- 'monthly'|'quarterly'|'yearly'
  period_key TEXT NOT NULL,   -- '2026-03'|'2026-Q1'|'2026'
  data TEXT NOT NULL,         -- JSON：报告全量数据
  is_read INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(user_id, type, period_key)
);
```

## 八、后端改动清单

| 新增文件 | 说明 |
|---------|------|
| `server/services/reportGen.js` | `generateMonthlyReport(userId, month)`、`generateQuarterlyReport`、`generateYearlyReport` |
| `server/routes/report.js` | `GET /api/report/list`、`GET /api/report/:id` |

> 不使用定时任务。报告在用户首次访问 `GET /api/report/list` 时按需生成：后端检查 reports 表中是否已有对应周期的缓存记录，若无则实时调用 `reportGen` 生成并写入缓存，若有则直接返回。
