## 一、背景与设计目标

### 为什么做修炼擂台

当前 App 中，属性的唯一消耗出口是打 Boss（愿望系统）。Boss 战的结果基本可以从属性值直接预测，缺少博弈感和意外性。

修炼擂台提供一种**以现实执行力为核心的竞技方式**：发起方出题，挑战方完成并提交实证，由发起方判定。竞争的不是属性高低，而是执行力、创意和诚信。

### 设计原则

- **与线下行为深度绑定**：擂台挑战的核心是完成一件真实的事，不是游戏内的数值比拼
- **发起方即裁判**：简化争议处理流程，同时通过质疑机制防止不公正判定
- **独立道具经济**：消耗擂台令（专属道具），不直接消耗属性道具，形成独立的参与成本

## 二、擂台令设计

### 2.1 擂台令是什么

**擂台令**是发起修炼擂台所需的专属道具，不参与炼化系统，只用于开启擂台。

修仙主题定名：**切磋令**（可选名：比武令、论道令）

外观：符篆形态，一次性使用。

### 2.2 切磋令的获取渠道

| 渠道 | 获取方式 | 数量 |
|------|---------|------|
| 每日签到 | 连续签到 7 天时额外获得 | 1 张 |
| 行为历练 | 当月首次完成某 sub_type 30 次（月度成就） | 1 张 |
| 秘境探索 | 探索等级提升时获得 | 1-2 张 |
| 特殊节气 | 冬至等特殊节气期间，行为打卡额外产出 | 1 张（活动期间） |

> 切磋令不可通过炼化或普通打卡获得，保持其稀缺性（每月大约可获取 2-4 张）。

## 三、擂台机制设计

### 3.1 发起擂台

消耗 1 张切磋令，发起方设定以下内容：

| 配置项 | 说明 |
|--------|------|
| 挑战内容 | 挑战方需要完成的任务描述（文字，必须具体可执行）|
| 时限 | 接受后，挑战方需在此时间内完成（可选：1天/3天/7天）|
| 难度 | 决定奖励等级（见3.5奖励设计）|
| 结束条件 | 先完成先赢 / 全员完成均可获奖 |
| 被挑战方 | 选择家庭内的成员（可多选，不可选自己）|
| 是否自我参与 | 发起方是否同时将自己设为挑战方（默认否）|

### 3.2 接受流程

- 被挑战方收到通知后，**24 小时内**决定是否接受
- 接受后，**立即开始计时**，计时基于各人接受的时间（非统一开始）
- 24 小时内未响应，视为拒绝，不计入挑战人数
- 所有被邀请方均拒绝 → 挑战令退回，擂台取消

### 3.3 提交与判定（正常流程）

```
挑战方在时限内完成任务
  → 提交照片（必须）+ 文字说明（可选）
  → 发起方收到提交通知
  → 发起方判定：通过 / 不通过（需在 24 小时内判定）
    ├─ 通过：该挑战方获胜，立即发放奖励
    └─ 不通过：该挑战方失败
  → 时限内未提交：自动判定失败
```

> 发起方未在 24 小时内判定 → 系统自动判定为通过（防止发起方拖延不判）

### 3.4 质疑机制

当发起方判定某挑战方"不通过"时，挑战方有权发起**质疑**：

```
挑战方发起质疑
  → 发起方被要求亲自完成同一挑战
  → 发起方需在"原时限相同的时间"内提交照片证明
  → 挑战方判定发起方提交是否通过
    ├─ 发起方通过：原判定有效，该挑战方失败（质疑失败）
    └─ 发起方不通过/超时：本场擂台无人获得奖励，切磋令消耗不退
```

**质疑规则补充**：
- 每场擂台每人只能质疑一次
- 质疑需在发起方作出判定后的 12 小时内发起
- 若多人同时质疑，发起方只需完成一次挑战，所有质疑方共同判定

### 3.5 所有挑战方失败时

当所有被挑战方均未能在时限内完成：

- **发起方获胜**，获得对应难度的奖励
- 若发起方自己也设为挑战方，则按正常流程参与判定

## 四、奖励设计

### 4.1 难度与奖励对照

| 难度 | 发起成本 | 挑战方获胜奖励 | 发起方全胜奖励 |
|------|---------|-------------|-------------|
| 入门 | 切磋令 ×1 | 良品道具 ×2（随机属性） | 良品道具 ×1 |
| 进阶 | 切磋令 ×1 | 上品道具 ×1 + 良品道具 ×1 | 上品道具 ×1 |
| 高阶 | 切磋令 ×1（消耗不变，但任务要求更高）| 上品道具 ×2 | 上品道具 ×1 + 良品道具 ×1 |

> 难度的差异不体现在切磋令消耗上（统一 1 张），而体现在发起方设定的任务难度和对应奖励。发起方设定高阶难度时，需同时准备更高价值的奖励道具。

**奖励道具的来源**：系统根据难度直接发放（不由发起方质押），奖励属于系统奖励，不消耗发起方库存。

> 这和家庭挑战赛的质押机制不同——修炼擂台的奖励是系统发放，发起方的"成本"只是切磋令。

### 4.2 "先完成先赢"模式下的多人处理

- 第一个提交被判定"通过"的挑战方获胜，立即发放奖励
- 之后其他挑战方仍可在时限内提交，但不再有奖励（除非模式为"全员完成均获奖"）
- 后续提交的人，发起方仍需判定，判定结果记录在案（影响质疑权利）

## 六、数据库设计

### 新增表：`arenas`（擂台主表）

```sql
CREATE TABLE arenas (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  family_id INTEGER NOT NULL,
  initiator_id INTEGER NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  difficulty TEXT NOT NULL,        -- 'entry'|'advanced'|'expert'
  time_limit_hours INTEGER NOT NULL, -- 挑战时限（小时）
  win_condition TEXT NOT NULL,     -- 'first_win'|'all_win'
  initiator_joined INTEGER DEFAULT 0, -- 发起方是否也参与
  status TEXT DEFAULT 'open',      -- 'open'|'active'|'judging'|'completed'|'cancelled'
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  expires_invite_at DATETIME NOT NULL -- created_at + 24小时
);
```

### 新增表：`arena_participants`（参与记录表）

```sql
CREATE TABLE arena_participants (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  arena_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  status TEXT DEFAULT 'invited', -- 'invited'|'accepted'|'declined'|'submitted'|'passed'|'failed'|'disputed'
  accepted_at DATETIME,
  deadline_at DATETIME,          -- accepted_at + time_limit_hours
  submission_text TEXT,
  submission_photo_url TEXT,
  submitted_at DATETIME,
  judged_at DATETIME,
  dispute_at DATETIME,
  UNIQUE(arena_id, user_id)
);
```

### 新增表：`arena_disputes`（质疑记录表）

```sql
CREATE TABLE arena_disputes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  arena_id INTEGER NOT NULL,
  disputant_id INTEGER NOT NULL,   -- 质疑发起人
  initiator_submission_text TEXT,
  initiator_submission_photo_url TEXT,
  initiator_submitted_at DATETIME,
  initiator_deadline_at DATETIME,
  verdict TEXT,                    -- 'initiator_pass'|'initiator_fail'|'pending'
  verdict_at DATETIME
);
```

## 八、待确认问题

1. 切磋令是否有过期机制？建议：无过期，可长期持有，但每月获取量有上限
2. 擂台是否支持"旁观"功能，家庭中未被邀请的成员能否查看进行中的擂台？建议：可查看，不可参与，增加观战趣味
3. 质疑环节发起方提交后，如何防止发起方提交低质量照片"糊弄过关"？建议：第一版由挑战方判定（接受人性风险），后续可增加家庭成员投票辅助
